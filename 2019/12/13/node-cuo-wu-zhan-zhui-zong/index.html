js的堆栈

首先我们需要知道，js的堆栈是什么，即js的函数执行是怎恶样的？
根据定义：堆栈的数据结构 就是后进先出，即 LIFO(last in, first out)

当函数调用时，他就会被压入堆栈的顶部，该函数运行完成之后，又会从堆栈顶部去除.

举个栗子🌰

function firstcall() {
  console.log('first');
  console.trace();
}
function secondCall() {
  console.log('second');
  firstcall();
}
function thirdCall() {
  console.log('third');
  secondCall();
}
thirdCall()

上述的函数调用 thirdCall最先调用（先入栈），然后thirdCall里面调用了secondCall 此时的secondCall入栈, 继而firstCall入栈,那么当firstCall执行完毕之后 出栈，随之secondCall, thirdCall依次出栈.

我们可以实际看一下具体的堆栈信息:

function firstcall() {
  console.log('first');
}
function secondCall() {
  console.log('second');
  firstcall();
  console.trace()
}
function thirdCall() {
  console.log('third');
  secondCall();
}
thirdCall()

以下为输出:

third
second
first
Trace
    at secondCall (test/test.js:7:11)
    at thirdCall (test/test.js:11:3)
    at Object.<anonymous> (test/test.js:13:1)
    at Module._compile (module.js:652:30)
    at Object.Module._extensions..js (module.js:663:10)
    at Module.load (module.js:565:32)
    at tryModuleLoad (module.js:505:12)
    at Function.Module._load (module.js:497:3)
    at Function.Module.runMain (module.js:693:10)
    at startup (bootstrap_node.js:188:16)

我们可以看到firstCall执行完毕之后就被推出栈了，所以看不到对应的堆栈追踪.

js中的Error类型

当我们写js的时候程序意外跑错或者异步函数错误捕获抛出，这个错误其实就是Error的实例，继承自js的Error类型.
但是js和node的Error类 是有些许区别的.

js的Error类型:

  • Error.prototype.constructor 原型的构造函数
  • Error.prototype.message 错误信息
  • Error.prototype.name 错误名字

没有对应的规范的stack和其他属性支持，详见mdn

nodejs的Error类型:

里面包含了丰富的错误时间和信息的捕获，这其中就包含了

  • new Error(message)
  • Error.captureStackTrace(targetObject[, constructorOpt])
  • Error.stackTraceLimit
  • error.code
  • error.message
  • error.stack

详情可参考nodejs的文档

对于node服务来说我们有时候需要详细的错误栈追踪来进行上报错误信息，或者进行日志的错误排查 都会需要用到，那么错误栈的追踪是怎么进行的，都包含了哪些内容呢?

Node.js 内置的 Error 类型有：

  1. Error：通用的错误类型，例如：new Error(‘error!!!’)。
  2. SyntaxError：语法错误，例如：require(‘vm’).runInThisContext(‘binary ! isNotOk’)。
  3. ReferenceError：引用错误，如引用一个未定义的变量，例如：doesNotExist。
  4. TypeError：类型错误，例如：require(‘url’).parse(() => {})。
  5. URIError：全局的 URI 处理函数抛出的错误，例如：encodeURI(‘\uD800’)。

每一个error的触发都会抱哈 对应的message，name，stack，constructor等信息，这些帮助我们更好的定位书写或者引用的其他库执行造成的错误信息，当然 这其中我们最想要知道的就是错误是在哪抛出来的。

stack trace

错误栈本质上其实就是调用栈,而调用栈我们上面已经先给出来了js的执行栈的情况。

我们先看一下js的错误常规捕获方式(同步函数):

为了抛出一个错误, 必须使用 throw 关键字. 为了 catch 一个抛出的错误, 必须使用 try…catch 包含可能抛出错误的代码. Catch的参数是被抛出的错误实例.

如 Java 一样, JavaScript 也允许在 try/catch 之后使用 finally 关键字. 在处理完错误之后, 可以在 finally 语句块作一些清除工作.

在语法上, 你可以使用 try 语句块而其后不必跟着 catch 语句块, 但必须跟着 finally 语句块. 这意味着有三种不同的 try 语句形式:

  • try…catch
  • try…finally
  • try…catch…finally
普通错误栈

抛出error类型的错误信息:

function runWithoutThrowing(func) {
    try {
        func();
    } catch (e) {
        console.log('There was an error, but I will not throw it.');
        console.log('The error\'s message was: ' + e.stack)
    }
}

function funcThatThrowsError() {
    throw new TypeError('I am a TypeError.');
}

runWithoutThrowing(funcThatThrowsError);

// 输出错误如下:

    at runWithoutThrowing (test/test.js:18:7)
    at Object.<anonymous> (test/test.js:29:1)
    at Module._compile (module.js:652:30)
    at Object.Module._extensions..js (module.js:663:10)
    at Module.load (module.js:565:32)

抛出一个string类型的（非error的类型）:

function runWithoutThrowing(func) {
    try {
        func();
    } catch (e) {
        console.log('There was an error, but I will not throw it.');
        console.log('The error\'s message was: ' + e.stack)
    }
}

function funcThatThrowsString() {
    throw 'I am a String.';
}

runWithoutThrowing(funcThatThrowsString);

// 输出结果如下
There was an error, but I will not throw it.
The error's message was: undefined

我们可以看到常规的抛错（error的类型）是可以追踪到具体的错误栈的，但是只是throw一个message的时候错误栈是没有的。

异步函数错误栈
  1. generator函数
const co = require('co');

function* functionOne() {
  yield new Promise((resolve) => {
    setTimeout(() => { resolve(); }, 1);
  });
  throw new Error('Something Bad');
}

function* functionTwo() {
  yield* functionOne();
}

co(functionTwo())
  .catch((error) => {
    console.log(error);
  });

  // 对应的输出
  Error: Something Bad
    at functionOne (test.js:7:9)
    at functionOne.next (<anonymous>)
    at functionTwo (test.js:11:10)
    at functionTwo.next (<anonymous>)
    at onFulfilled (test.js:65:19)

对于generator函数来说，对应的错误栈是明确的。

  1. async函数

但是对应相关异步函数的深度调用，错误栈又是怎么样的呢？

我们来看几个🌰：

async function functionOne() {
  await new Promise((resolve) => {
    setTimeout(() => { resolve(); }, 1);
  });
  throw new Error('Something Bad');
}

async function functionTwo() {
  await functionOne();
}
functionTwo()
  .catch((error) => {
    console.error(error);
  });

  // 对应的输出结果如下:
  Error: Something Bad
    at functionOne (test/test.js:5:9)
    at <anonymous>

可以看出上面的错误栈只是追踪到了functionOne，对于调用其的functionTwo并没有显示在对应的错误栈中。

但是如果抛错是在任何一个await之前执行的呢？

async function functionOne() {
  throw new Error('Something Bad');
}

async function functionTwo() {
  await functionOne();
}
functionTwo()
  .catch((error) => {
    console.error(error);
  });

  // 错误栈如下
  Error: Something Bad
    at functionOne (test/test.js:5:9)
    at functionTwo (test/test.js:9:9)
    at Object.<anonymous> (test/test.js:11:1)

可以看到当抛错程序之前没有await执行时，错误栈的信息是完整的

引用nodejs的贡献者之一的话是:

Generators are resumed from user code. Async functions are resumed from the micro task queue.

generator函数执行结束依旧回到用户的代码片段，但是async函数执行结束时回到了对应的微任务队列

如何优雅的使用async函数

经过上面的有关的错误栈的捕获，我们都知道对一个async函数的错误捕获需要使用try/catch，但是如果try/catch写的过多就会引起代码重复，并且美观上也会有所欠缺，所以。。。

  1. 所以我们可以通过函数来替代try/catch 错误捕获

类似于toawait使用ts来表现:

export function toWait<T, U = Error> (
  promise: Promise<T>,
  errorExt?: object
): Promise<[U | null, T | undefined]> {
  return promise
    .then<[null, T]>((data: T) => [null, data])
    .catch<[U, undefined]>((err: U) => {
      if (errorExt) {
        Object.assign(err, errorExt);
      }

      return [err, undefined];
    });
}

对应的错误可以不用写过多的try/catch来进行错误捕获

  1. 尽可能少使用async函数

我们在上述的v8文档中可以看出来async函数的消耗还是比较多的，相比于promise和其他的user code。所以 请慎用async函数
